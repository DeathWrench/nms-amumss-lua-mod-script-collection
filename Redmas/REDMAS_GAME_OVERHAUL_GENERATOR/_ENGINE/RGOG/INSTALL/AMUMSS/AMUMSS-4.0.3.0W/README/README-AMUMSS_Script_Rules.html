Which parts of LUA language can I use in scripts? - Back - Top

    All parts of LUA 5.4 The Language are available.
    
    Most parts of LUA 5.4 The Standard Librairies are available as listed here:  
        - All available standard libraries in {string, math, table}
        - io.open, io.input, io.type, io.read, io.lines and io.close
        - os.clock, os.date, os.difftime, os.getenv, os.time, os.tmpname
        - assert, coroutine, getmetatable, ipairs, next, pairs, pcall, print, select,
          setmetatable, tonumber, tostring, type, xpcall

    In addition:
        - lfs.currentdir, lfs.attributes and lfs.dir from LuaFileSystem Reference
        
        - GUIF({default_value as string|number|bool, prompt as short|long string}) (a.k.a. Get_User_Input_Function).  
            An internal function that can be used to request information from the script user while it is processed.
            >>> GUIF() (click for more info) can be used in a script body AND in VCT of EXML_CHANGE_TABLE 
            
            GUIF() will return a value of the same type() as 'default_value' is
                * no user input and just 'ENTER' key will return the default value
                * if the input is not of the type required, the default value is returned
                * 'bool' will return a boolean
                      will ask for Y/N like: Enter Y/N:
                      but User input can be any of (0/n/N/false/FALSE or 1/y/Y/true/TRUE)
                      if they wish (case does not matter)
                * 'number' will return a number
                      will ask for a number like: Enter Num:
                * 'string' will return a string
                      will ask for a string like: Enter Str:

MOD_FILENAME - Back - Top

    - member of NMS_MOD_DEFINITION_CONTAINER    
    - always REQUIRED  
    
    - a STRING, the name of the pak file created by the tool  
        ex.:    ["MOD_FILENAME"]        = "some.pak",  
        (unless a combined mod is created, then the name may not be used)  
    - if MOD_FILENAME is not found, it will default to 'GENERIC.pak' and a [WARNING] will be issued  
    - maximum name length is 106 char + .pak

GLOBAL_INTEGER_TO_FLOAT - Back - Top

    - member of NMS_MOD_DEFINITION_CONTAINER    
    - OPTIONAL  
    
    - a STRING  
  
    ***** USE 'GLOBAL_INTEGER_TO_FLOAT' ONLY if you know that changing all INTEGERs to FLOAT is right  
        when the math operation results in a FLOAT  
      
    - Note: INTEGER_TO_FLOAT will supersede GLOBAL_INTEGER_TO_FLOAT if present  
  
    - [WARNING] even if you use this option to silence the AMUMSS WARNING/NOTICE's,  
        MBINCompiler may still refuse to compile the EXML when the underlying value is 'really' an INTEGER.  
  
    - '[WARNING] ORIGINAL and NEW number value have mismatched types (INTEGER vs FLOAT or STRING vs NUMBER)'  
        You may see this WARNING in the REPORT.   
        This WARNING only tells you that the script is changing an INTEGER "0" to a FLOAT "0.75"  
            or a STRING to a NUMBER or a NUMBER to a STRING.  
        If the EXML compiles correctly with MBINCompiler then it is alright,   
            otherwise it may be the case why MBINCompiler fails to compile the EXML.    
        So it is an AID to you in finding a possible problem.  
      
    - can have one of these values: {nil, "", "FORCE", "PRESERVE"}  
    - default value is ""  
    - ["GLOBAL_INTEGER_TO_FLOAT"] = "FORCE", will change the INTEGER value to a FLOAT if necessary  
    - ["GLOBAL_INTEGER_TO_FLOAT"] = "PRESERVE", will keep the same type of value as before the operation  
        and hide: "[WARNING] ORIGINAL value below is INTEGER."   
        >> A FLOAT will still be a FLOAT  
        >> An INTEGER will still be an INTEGER (rounding up the value if necessary)

AMUMSS_SUPPRESS_MSG - Back - Top

    - member of NMS_MOD_DEFINITION_CONTAINER    
    - OPTIONAL  
    - case-insensitive  

    - a STRING, the name of the message to SUPPRESS   
        ex.:    ["AMUMSS_SUPPRESS_MSG"] = "SUPPRESS_FLOOR_DIV",  -- must be on one line   
                ["AMUMSS_SUPPRESS_MSG"] = "anotherOne",  -- must be on one line   
                ...
    - if found, it will SUPPRESS the message in the log.lua file AND in the REPORT.lua file  
    
    - multiple ["AMUMSS_SUPPRESS_MSG"] are allowed
    - entries must be on one line as shown above and below and inside NMS_MOD_DEFINITION_CONTAINER table
    - values must be exactly the STRING shown below, not a variable with that value
    - current valid values are:  
        * ["AMUMSS_SUPPRESS_MSG"] = "SUPPRESS_FLOOR_DIV", --(notice regarding '//', floor division operator)   
        * ["AMUMSS_SUPPRESS_MSG"] = "MULTIPLE_STATEMENTS", --(notice regarding multiple_statements on same line)

MBIN_FILE_SOURCE - Back - Top

    - member of MBIN_CHANGE_TABLE    
    - REQUIRED  
    - alias is "MBIN_FS" (must be uppercase)  
    
    - Alternate syntax #1: a NMS MBIN path STRING,  
          as in:
          [[METADATA\SIMULATION\SCENE\EXPERIENCESPAWNTABLE.MBIN]],  
        - used to work on this one file  
  
    - Alternate syntax #2: a TABLE of NMS MBIN path STRINGS, as in  
        {  
          [[METADATA\SIMULATION\SOLARSYSTEM\BIOMES\BARREN\BARRENHQOBJECTSFULL.MBIN]],  
          [[METADATA\SIMULATION\SOLARSYSTEM\BIOMES\BARREN\BARRENOBJECTSDEAD.MBIN]],  
          [[METADATA\SIMULATION\SOLARSYSTEM\BIOMES\BARREN\BARRENOBJECTSFULL.MBIN]],  
          ...  
        },  
        - used to work on all those files from top to bottom  
  
    - Alternate syntax #3: a TABLE of sub-table(s) of NMS_MBIN_PATH_FILENAMES STRINGS,  
        NEW_PATH_FILENAMES STRINGS and an Optional_Flag  
        see 'NEW_PathFilename.lua' for an example of use, as in


    {  
      {[[METADATA\SIMULATION\SOLARSYSTEM\BIOMES\BARREN\BARRENOBJECTSFULL.MBIN]], [[NEWPATH\temp1\My.GLOBALS.MBIN]], Optional_Flag},  
        ...
    },

        - this allows you to create a new MBIN file (My.GLOBALS.MBIN) in any folder you like (NEWPATH\temp1\)  
          BASED on an original NMS pak file (BARRENOBJECTSFULL.MBIN)  
  
        Note #1: The create/rename part happens before processing the whole MBIN_FILE_SOURCE section  
            - the My.GLOBALS.MBIN file will be identical to the original BARRENOBJECTSFULL.MBIN file  
            - the BARRENOBJECTSFULL.MBIN file will be modified by the content  
              of the enclosed EXML_CHANGE_TABLE unless the flag "REMOVE" is used. 
            - if you want to modify the new My.GLOBALS.MBIN file,  
              you need to add a new section in MBIN_CHANGE_TABLE  
              with MBIN_FILE_SOURCE now referencing the new My.GLOBALS.MBIN file  
              (see NEW_PathFilename.lua for an example of use)  
  
        Note #2: Optional_Flag can be omitted or ""  
            - if Optional_Flag is "REMOVE"  
              then the original file BARRENOBJECTSFULL.MBIN will NOT be included in the mod pak
              and becomes unavailable for further processing.
              (do not REMOVE it if you want to use it in further script actions)

SPECIAL_KEY_WORDS - Back - Top

    - member of EXML_CHANGE_TABLE    
    - OPTIONAL  
    - alias is "SKW" (must be uppercase)  
     
    **** It is HIGHLY suggested to use MapFileTrees files to easily locate valid SPECIAL_KEY_WORDS ****
     
    - a TABLE of (1+)(two STRING pairs)  
    - KEYWORD STRINGs are case-insensitive  
    - "", or {}, or {"",}, have no effect ==> all these mean: no SPECIAL_KEY_WORDS to search  
      (the whole file is used if nothing else selects a section)  
            ex.: ["SPECIAL_KEY_WORDS"] = {"PropertyName1","value1","PropertyName2","value2",...},  
              
    - "Propertyname_1","value_1", ==> SPECIAL_KEY_WORDS pair #1  
      "Propertyname_2","value_2", ==> SPECIAL_KEY_WORDS pair #2
      ...
        that will be used (optionally with the PRECEDING_KEY_WORDS) to help define the SECTION(s)  
        we are looking for  
    - One or more pairs can be used of the form {"Propertyname_1","value_1","Propertyname_2","value_2",...},  
    - "Propertyname_1","value_1" defined a SECTION  
        then "Propertyname_2","value_2" defined a sub-SECTION of SECTION  
        and so on until the final sub-Section is found  
    - "Propertyname_X" and/or "value_X" can have the value "IGNORE"  
        if that helps selecting the right section (use with care!)  
    - "value_2" here (or the last of the "values" when multiple pairs are used) can be "IGNORE"  
        (useful in the case where NMS changes that value later)  
        see the script LearnMoreWords.lua as an example...  
    - If the line pointed to by the SPECIAL_KEY_WORDS pair(s) is found,  
        it will be used by LINE_OFFSET (if any)  
  
    - WHAT ARE CONSIDERED SPECIAL_KEY_WORDS?
        =>> Have a look at the MapFileTrees folder for the file matching the EXML 
        * any line with a 'Property' AND a 'value'  
        * where the value is NOT ""  
        * lines with a value like a filename (???.MBIN or ???.xml for example)  
          are accepted as SPECIAL_KEY_WORDS  
            **** be warned that filenames like ???.MBIN could change more often  
                 and render your script obsolete!  
        * lines with a value like 'True', 'False' or a number are accepted as SPECIAL_KEY_WORDS  
            **** be warned that these types SHOULD be used with EXTREME CAUTION,  
                 HG could easily change the value!!!  
      
    - Works alone or with PRECEDING_KEY_WORDS (see OPTIONAL use below and MiningLaserSpeed.lua for example)  
    - If SPECIAL_KEY_WORDS are used, they are processed BEFORE the PRECEDING_KEY_WORDS  
        unless ["PRECEDING_FIRST"] = "TRUE"

PRECEDING_FIRST - Back - Top

    - member of EXML_CHANGE_TABLE    
    - OPTIONAL  
     
    - a STRING  
    - DEFAULT value is "FALSE"  
    - can have one of these values: {nil, "", "TRUE", "FALSE"}  
            ex.: ["PRECEDING_FIRST"] = "TRUE",  
    - "TRUE" means: PRECEDING_KEY_WORDS will be processed 'before' SPECIAL_KEY_WORDS  
    - nil, "" or "FALSE" mean: PRECEDING_KEY_WORDS will be  
      processed 'after' SPECIAL_KEY_WORDS (the default behavior)

PRECEDING_KEY_WORDS - Back - Top

    - member of EXML_CHANGE_TABLE    
    - OPTIONAL  
    - alias is "PKW" (must be uppercase)  
    
    **** It is HIGHLY suggested to use MapFileTrees files to easily locate valid PRECEDING_KEY_WORDS ****
     
    - a STRING or (TABLE of STRINGs)  
    - KEYWORD STRINGs are case-insensitive  
    - "", or {}, or {"",}, have no effect ==> all these mean: no PRECEDING_KEY_WORDS to search  
     (the whole file is used if nothing else selects a section)  
      
    - {"key1","key2","key3",...}, ==> are a list of key_words  
            (only 'Property name=' or 'Property value=') inside an EXML  
            ex.: ["PRECEDING_KEY_WORDS"] = {"PropertyNameORPropertyValue1","PropertyNameORPropertyValue2",...},  
              
    - LEVEL 0 being the whole EXML file SECTION  
    - LEVEL key1, key2, ... defined SECTION and sub-SECTION heads (SECTIONs) in the EXML   
        file hierarchy (lines that end in [">] ) UNLESS used like in 'ALTERNATE USE' below  
        
    - Keys always refer to same LEVEL or sub-LEVEL keys   
    - Keys NEVER go back up the LEVEL chain
    - Skipping one or more LEVEL can work, specially if you want to specify many sections (see also REPLACE_TYPE)  
    
    - PRECEDING_KEY_WORDS works alone and with SPECIAL_KEY_WORDS  
    - If SPECIAL_KEY_WORDS are used, they are processed BEFORE the PRECEDING_KEY_WORDS  
        unless ["PRECEDING_FIRST"] = "TRUE"  
          
    - In NORMAL use: (see also ALTERNATE USE)

    - alias is "WIS" (must be uppercase)

                 >>> if any one pair is found then the section is retained

        A) IF none of the "Property name/value" with that "WhereValue" is found in the section   
            - then that SECTION is skipped/ignored  
        B) IF one or more "Property name/value" have that "WhereValue" in the section   
            - then that SECTION will be used by the script to update the EXML

                 >>> if all pairs are found then the section is retained

        A) IF all of the "Property name/value" with those "WhereValue" are found in the section   
            - then that SECTION will be used by the script to update the EXML  
        B) IF just one "Property name/value" does not have that "WhereValue" in the section   
            - then that SECTION is skipped/ignored  
    
    - WHERE_IN_SECTION is processed before WHERE_IN_SUBSECTION if also present  
              
    - "IGNORE" can be used in place of one or both strings

MATH_OPERATION - Back - Top

    - member of EXML_CHANGE_TABLE    
    - OPTIONAL  
    - alias is "MATH_OP" (must be uppercase)  
    
    - "" or a STRING composed of  
    - == "$"(OPTIONAL) + one of {+, -, *, /, //, %, ^}
    - == SUFFIX (see below)
        
        ex.:    ["MATH_OPERATION"]  = "*",  
            or  ["MATH_OPERATION"]  = "$-",  
            or  ["MATH_OPERATION"]  = "*F:MaxAmount",  
            or  ["MATH_OPERATION"]  = "!$*L:5",  
  
    - "$" when used, commutes the order of the operands so that,  
         for example: (3 - 5 = -2, 'negative' 2) becomes (5 - 3 = 2, 'positive' 2)  
  
    MATH_OPERATION is applied 'after retrieving a LINE' pointed to by the KEYWORDS and   
                   the VALUE_CHANGE_TABLE (VCT) Property we are looking for and  
                   possibly redirected by LINE_OFFSET  

          Note: the value pointed to MUST be a numerical value, otherwise a value of 0 will be used  
              (so NOT a text string like [ALLOY1] here <Property name="Value" value="ALLOY1" />)  
          
    - "$"(OPTIONAL) + one of {+, -, *, /, //, %, ^} used alone will apply the math operation  
         between the VALUE at that LINE and the script VALUE_CHANGE_TABLE "newvalue".  
         Then use it as the mod "newvalue"  
  
    >>> [WARNING] {F:, FB:} is probably more robust to NMS changing things than {L:, LB:}  
                  With {L:, LB:}, a one line added/removed by NMS <<in between>> would prevent the expected change  

    - with SUFFIX == "!"(OPTIONAL) + "$"(OPTIONAL) + (one of {+, -, *, /, //, %, ^}) + (one of {F:, FB:})   
            + an endString ("IGNORE" or a Property Name STRING like [MaxAmount] 
            in this line: <Property name="MaxAmount" value="300" />)  
  
          F: == Fetch Forward (after) the current LINE for endString and get the value on that new line  
         FB: == Fetch Backward (before) the current LINE for endString and get the value on that new line  
  
         "!", when used, locks in the retrieved value to the first one found to be used in the calculation
         
         Now apply the math operation between this value and the script "newvalue".  
         Then use it as the mod "newvalue".  
  
    - with SUFFIX == "!"(OPTIONAL) + "$"(OPTIONAL) + (one of {+, -, *, /, //, %, ^}) + (one of {L:, LB:}) + a NUMBER_OF_LINES  

          L: == Lookup Forward (after) the current LINE PLUS NUMBER_OF_LINES and get the value on that new line  
         LB: == Lookup Backward (before) the current LINE MINUS NUMBER_OF_LINES and get the value on that new line  
  
         NUMBER_OF_LINES should be calculated from the first NUMBER in the section (may not be the top of the section)  
         
         "!", when used, locks in the retrieved value to the first one found to be used in the calculation
         
         Now apply the math operation between this value and the script "newvalue".  
         Then use it as the mod "newvalue".

    - see 'Test_VALUE_MATCH Patterns.lua' as an example

VALUE_CHANGE_TABLE - Back - Top

    - member of EXML_CHANGE_TABLE    
    - OPTIONAL  
    - alias is "VCT" (must be uppercase)  
    
    - a TABLE of (1+)sub-table(s) of (two STRINGs)  
        ex.:  
            ["VALUE_CHANGE_TABLE"]  = {  
                    {"Property name/value1","newvalue1",},  
                    {"Property name/value2","newvalue2",},  
                    {"Property name/value3","newvalue3",},  
                    ...  
            },  
    
    - in other word, a table containing one or more "two string" tables describing:  
        - on the left side: the "Property name=" OR "Property value=" we want to change the 'currentvalue' of  
        - and, on the right side, a "newvalue" that will replace the original one in the EXML file  
    
    - For STRING newvalues only:  
      * A 'newvalue' of the form "Something{:}Somethingelse" will replace the 'Original Value' with
        'SomethingOriginal ValueSomethingelse'
        In other words, the 'Original Value' is prefixed with the left side of {:} and suffixed with the right side of {:}.
        One or both sides can be an empty string (both sides empty is equivalent to keeping the 'Original Value' intact)
    
    **** If you need more information, please see "Property name/value1" **** 
    
    - "Property name/value" are case-insensitive  
    - "Property name/value" can be a 'regular string expression' (as defined by LUA 5.4 Patterns)  
      A 'regular string expression' is enclosed in curly brackets {...},  
      and will match "Property name/value" we are looking for  
    
    - if all "Property name/value" are the same and not 'IGNORE'   
      then AMUMSS will adjust the next line automatically (including LINE_OFFSET, if used)   
      in order to continue replacing down the section/file
    
        [WARNING] For ex.: "Bonus" here will trigger AMUMSS automatic line_adjustement  
                        ["VALUE_CHANGE_TABLE"]  = {  
                                {"Bonus","100",},  
                                {"Bonus","50",},  
                                {"Cost","25",},  -- bad idea, AMUMSS will complain!
                        },
                  
                  This could lead to unreliable replacements!  
                  Do not mix/use different "Property name/values""  
                  Please, in this case, split into two EXML_CHANGE_TABLE sub-tables instead
                      using the same KEYWORDS and other commands in both except the VALUE_CHANGE_TABLE

    - one or both STRING can be "IGNORE" in some cases (like when we use LINE_OFFSET)  
        - syntax examples:  {"Bonus","IGNORE",}, or  {"IGNORE","IGNORE",}, or  {"IGNORE","100",},  
            * {"Bonus","IGNORE",}, makes AMUMSS goto and SKIP the line with "Property name/value" == "Bonus",  
            * {"IGNORE","IGNORE",}, makes AMUMSS SKIP the next line,  
                doing NO exchange at all and continuing processing the next "Property name/value"  
            * {"IGNORE","100",}, makes AMUMSS change the line "value" to "100" regardless of the "Property"  
        
    - to update a group of numerical 'Property value=' using (for example) a ["MATH_OPERATION"] = "*"  
        and ["VALUE_CHANGE_TABLE"] = {{"IGNORE","1.5"},},  
        to multiply << each >> value by 1.5 without the need for ["LINE_OFFSET"]

How to allow 'user input requests' when processing scripts - Back - Top

        >>> script 'Test_GUIF_VCT.lua' is a good example to better understand what follows...
              
        NOTE: use OPTION '-DoConfigGlobal' (it can be set in BUILDMOD_AUTO.bat, default is 'Y')
              to control if, globally, user input requests are allowed when processing scripts,
              or answer the question about it when AMUMSS asks
              
              When true, all scripts are allowed to control their own requests for user input
              When false, all scripts requests for user input are disabled
              
        - GUIF({default_value as string|number|bool, prompt as short|long string}) (Get_User_Input_Function).  
            An internal function that can be used to request information from the script user while it is processed.
            >>> GUIF() can be used in a script body AND in VCT of EXML_CHANGE_TABLE 
            
            GUIF() will return a value of the same type() as 'default_value' is
                * no user input and just 'ENTER' key will return the default value
                * if the input is not of the type required, the default value is returned
                * 'bool' will return a boolean
                      will ask for Y/N like: Enter Y/N:
                      User input can be any of (n/N or y/Y), case does not matter
                * 'number' will return a number
                      will ask for a number like: Enter Num:
                * 'string' will return a string
                      will ask for a string like: Enter Str:
        
        - Example #1: usage of GUIF() in the script body (outside of NMS_MOD_DEFINITION_CONTAINER):
            NOTE: in the body of a script, GUIF() is used like any other function in the script

                var1 = 1 -- the name and default value of some variable
                
                GUI = { -- a table named GUI (or something else)
                        
                        -- a 'boolean' variable named DoConfig (or something else)
                        --    with a default value 'false'
                        --    and a single-line prompt to show the script user at processing time
                        DoConfig = { false, [[Would you like to edit this script? ]] },
                                              
                        -- a 'number' variable (the variable 'var1' assigned a value of 1 above)
                        --    with the default value '1' as a number
                        --    and a multi-line prompt
                        var1 = { var1, [[var1 (default: ]]..var1..[[) 
                        Enter a number >= 0.1: 
                                              ]] },
                                              
                        -- other variables to configure
                    }
                
                DoConfig = GUIF( GUI.DoConfig )
                if DoConfig then
                    var1 = GUIF( GUI.var1 )
                    -- other variables to configure
                end
                
                -- as the author of the script,
                --       you are responsible to check if the entered values are acceptable
                assert(type(var1) == "number" and var1 >= 0.1,
                    "Invalid var1 defined: Must be a numeric value of at least 0.1")
                
                -- var1 can be used as usual
                if var1 ~= 1 then
                -- do something here
                end

        - Example #2: usage of GUIF() in VCT of EXML_CHANGE_TABLE:
            NOTE: in VCT, GUIF() is defined in a short/long string
                  (use the same syntax as above in the GUI table above,
                       just make it a 'string' short or long as needed)
                
                EXML_CHANGE_TABLE  = {

                      ]=]} )]], },                                           -- asking for user input

                    },
                  },
                },

Meaning of "Property name/value" - Back - Top


    In this document, "Property name/value" means the name of a "Property name="   
        OR the value of a "Property value=" in the EXML files.  
    
    "value" here does NOT refer to the "value=" part   
        of lines like <Property name="ThisPropertyNAME" value="ThisPropertyValue" />  
    
    For example, if a fictional EXML contains:
        ...
        <Property name="Name" value="TEMPLATE_NAME" />   -- Case 1
        <Property name="ActiveLodDistances">             -- Case 2
          <Property value="TkLODDistances.xml">          -- Case 2
            <Property name="Distances">                  -- Case 2
              <Property value="0" />                     -- Case 3
              <Property value="10" />                    -- Case 3
              <Property value="40" />                    -- Case 3
            </Property>                    -- never used
          </Property>                      -- never used
          <Property name="List" />                       -- Case 4
          ...
        </Property>                        -- never used
        ...
            
    Case 1:          <Property name="Name" value="TEMPLATE_NAME" />
        ->  "Property name" is named "Name"       and has a "value=" of "TEMPLATE_NAME"
        ->  The left-side of VALUE_CHANGE_TABLE would be "Name", the Property to change
            and the "newvalue" would replace "TEMPLATE_NAME"  
        ->  This type of "Property name/value" can be used with SPECIAL_KEY_WORDS
        
    Case 2:          <Property value="GcStatsBonus.xml">
                 or  <Property value="TkLODDistances.xml">
                 or  <Property name="Distances">
        ->  These have "Property name/value" associated with them
            like "GcStatsBonus.xml", "TkLODDistances.xml" and "Distances"  
            but they do not have a "value=" associated with them
        ->  This type of "Property name/value" cannot be used with VALUE_CHANGE_TABLE  
            They can be used with PRECEDING_KEY_WORDS and VALUE_MATCH  
        
    Case 3:          <Property value="0" />
                 or  <Property value="10">
                 or  <Property value="40">
        ->  Have no "Property name/value"             but have "value=" of "0", "10" or "40"   
        ->  Having no "Property name", we substitute "IGNORE" for the left-side   
            and the right-side "newvalue" would replace current value "0", "10" or "40" in a VALUE_CHANGE_TABLE  
            depending which line is current based on the script commands
        ->  VALUE_MATCH can use them

    Case 4:          <Property name="List" />
        ->  This should not be used in a VALUE_CHANGE_TABLE  
        ->  If found, it usually mean an empty "List" and we could create the "List" using ADD  
        
    So, when we say: (using the EXML lines above)  
            ["VALUE_CHANGE_TABLE"]  = {  
                {"Property name/value1","newvalue1",},  
                {"Property name/value2","newvalue2",},  
            },  
            
        we mean (based on above examples):  
            ["VALUE_CHANGE_TABLE"]  = {  
                {"Name","MyTEMPLATE_NAME",},  -- where "MyTEMPLATE_NAME" is the "newvalue1"
                                              -- that will replace "TEMPLATE_NAME"
            },  
        and (based on above examples):    
            ["VALUE_CHANGE_TABLE"]  = {  
                {"IGNORE","35",},             -- where "35" is the "newvalue2" that will replace the current value 
                                              -- at the current line based on the script commands
            },

    How to disable ModScript scripts in sub-folder: ___DONOTUSE.txt - Back - Top

    * Disabling a scripts in a ModScript sub-folder means to instruct AMUMSS to IGNORE those files in the sub-folder.

    * A '___DONOTUSE.txt' file present in a sub-folder of ModScript will instruct AMUMSS to IGNORE that sub-folder and ALL its content: files and sub-folders.

    * Renaming the file to '___USE.txt' will instruct AMUMSS to USE that sub-folder and ALL its content (files and sub-folders) when processing.

    So, no need anymore to move sub-folders of ModScript out of ModScript to de-active them. Just create/paste a file named '___DONOTUSE.txt' in the sub-folder and AMUMSS ignores ALL its content.

    * An alternate solution is to drop files or folders into the 'Disabled scripts and paks' folder. '___DONOTUSE.txt' is not required in this special 'Disabled scripts and paks' folder.

    * BEST alternate solution is to use the app ModScript_Manager.exe to handle this process.

    Using the COMBINE_FLAG file: ___COMBINE.txt - Back - Top

    * The presence of a file called '___COMBINE.txt' in ModScript main folder instructs AMUMSS to ONLY auto-combine the scripts in ModScript main folder, overriding the user choice for INDIVIDUAL paks.

    To combine ALL scripts in ModScript AND sub-folders, the user has to make the choice to COMBINE.

    * The presence of a file called '___COMBINE.txt' in a sub-folder of ModScripts instructs AMUMSS to auto-combine the scripts and 'MEFTI' files in this sub-folder, overriding the user choice for INDIVIDUAL or COMBINE MOD paks.

    * In other words: When you, as a 'modder', use ___COMBINE.txt in a sub-folder, you are saying these scripts need to be combined to work regardless of the choice the user makes for the other scripts in its ModScript folder and sub-folders.

    * You can de-active the auto-combine process by renaming the file as ___COMBINEx.txt in the sub-folder

    * BEST alternate solution is to use the app ModScript_Manager.exe to handle this process.

    MapFileTrees - Back - Top

    NOTE: MapFileTree files are only updated when you run a script that opens the MBIN file >>> So run a script first like ModScriptCollection\MapFileTree_UPDATER.lua to update them when NMS updates. (even if it is only with, for example: ["MBIN_FILE_SOURCE"] = "METADATA/REALITY/TABLES/COSTTABLE.MBIN",)

    MapFileTree are helper files. If you have a look in the MapFileTrees folder, you should be able to find files that describe the SPECIAL (S) and PRECEDING (P) pairs/words that AMUMSS understand as KEYWORDS for a given EXML file.

    The name of the file is made up of the NMS path to that file (separated by dots) + the name of the EXML.lua (to provide the automatic folding capability).

    When UNIQUE, a KEYWORD pairs is shown with (SU). A UNIQUE is ONLY found ONCE in an EXML file.

    Know that a (S) can become a (SU) inside a section when used as a 2nd, 3rd, ... pair to further zoom in on a sub-section.

    LANGUAGE files are especially processed to reduce the line count to 5.5% compared to the original line count (like 30706/552642 lines) and to speed up their creation.

    AMUMSS 'Order of Processing' and NMS 'Load Order': 
    * From top to bottom (bottom mod pak is loaded last thus wins)
    * Note: not all first characters where tested here

            -  ALT+45  (a minus or dash)
            .

            9.

            A
            a
              ...
            Z
            z
            _  ALT+95 (underscore)
            __ ALT+95 (underscore x2)
        
    IMPORTANT NMS MODS processing by NMS.exe:
        Do not confuse "load order" and "which mod is the winner", two different things.

        Remember: If 2 mods modify the SAME file (EXML), the last mod to load THAT file wins
        (overwrites the EXML file loaded before) (see also 'NMS Load Order' above).  

        Which means that some (or all) of the files from an earlier loaded mod are used 
        and some (or all) of the files from a later loaded mod are also used
        when they are not the same exact files by name (.MBIN, .DDS, ...).

        BUT if any are the same files, the last mod to load that or those particular file(s)
        always 'wins the contest' and overwrites these individual files to be used by NMS.

    About MOD pak file extension (the .pak at the end of the mod file) - Back - Top

        Mod paks do not need to end with the extension '.pak' but must have '.pak' in the filename 
    to be recognized as a pak by NMS.  
    
    ANY supplemental extension could be added after the .pak and could be used to group together mods  
    from an author when viewed in Windows Explorer (using the column 'Type')
    (independently from the name of the paks which dictates the Load Order)
    
    ex.: MyNewMod.pak.wbertro is quite fine as a mod name AND does NOT disable the mod
    
    Also MyNewMod.pakwbertro also works AND does NOT disable the mod
      but MyNewMod.Xpakwbertro does disable the mod (notice that the extension now starts with .Xpak not .pak) 
            ('X' or anything else could be used to make '.pak' not a NMS mod)

    What AMUMSS will do... - Back - Top

        While following the 'Load Order', AMUMSS will do...
    
    - When only paks are in 'ModScript':
        * each pak is unpacked/decompiled and saved in its own folder in 'UNPACKED_DECOMPILED_PAKs'
    
    - When only ONE pak is in 'ModScript':
        * the pak is unpacked/decompiled and saved in its own folder in 'UNPACKED_DECOMPILED_PAKs'
        * If a .lua script is found in the PAK, you are asked if you want to rebuild a MOD from
          that script...
    
    - When only ONE or MORE .lua scripts are in 'ModScript':
        * If a 'combined' pak is requested, all the scripts will be used to create the pak mod
        * If an 'individual' mod is choosen instead, each script will create it owns pak mod
        
    - When ONE or MORE .lua scripts with ONE or MORE paks are in 'ModScript'...
        * the scripts will look into each pak to try locating the required MBINs and, if found, 
          use them.  Otherwise, NMS MBINs will be used
        * you can create Patch mods with this
        
            ========================= WARNING ========================

    Having more than one pak in ModScript will NOT combine both paks if they modify the same MBIN file,
    only the last MBIN loaded will combine with the script(s)
          
    - AMUMSS can handle correctly multiple paks in ModScript
      but be aware that only the last same mbin file loaded will be the one used by the script(s)
        
            ================== VERY IMPORTANT WARNING ================

    Using an 'older' pak (a pak that is NOT updated to the 'current' NMS MBIN file content)  
    could result in a not functional pak that could crash NMS.

    - Since AMUMSS will fetch the MBIN (called for by the script(s)) from the 'older' pak, 
      if that MBIN is NOT a modded 'current' version, the resulting modded MBIN may not be
      compatible with NMS
        
    - Note that AMUMSS cannot correct the faulty MBIN when it is an older version because it cannot
      know if the changes made in the 'older' pak to the MBIN were intentional (by the modder)
      or due to NMS upgrading that MBIN

    - Back - Top
    
    - When only MBINs are in 'ModScript':
        * The .MBINs are passed to the appropriate MBINCompiler.exe to decompile them to .EXML
        
    - When EXMLs are in 'ModScript':
        * NOTE: This feature has not been fully tested in all situation and could have side-effects
        * With scripts: if the EXMLs are in proper NMS / script folders as indicated by the script,  
          the .EXMLs are used in place of NMS / mod pak decompiled MBINs  
          These ModScript EXMLs have priority over NMS pak ones!!!

        depending on your OPTIONS, you will be asked if you want to include them in the final PAK   
        All created paks will contain these files if you answer 'Y'

    - If any files are in the sub-folder 'MEFTI' of your custom script folder,
        they will be automatically included in the final PAK.

        A custom script folder is a sub-folder of ModScript that includes ALL parts 
        of your mod (.lua script and where extra files are in its own 'MEFTI' folder)
        Thus making it easy to create a full package to distribute

        for the scripts in that sub-folder and end up in the mod

        then these 'MEFTI' folders will also contribute to the final combine mod pak

    - Where do MBINs/EXMLs come from?
        Note: .MBIN are files found inside .pak files
                and we access them by unpacking the .pak file
        
        * If a .MBIN required by a script is already in 'MODBUILDER\MOD', it is used
        
        * ElseIf some .pak(s) is/are present in 'Modscript':
            First look for the .MBIN in all the paks (last pak loaded with the .MBIN wins)
            and then in NMS PCBANKS if not found

        * Else (no .pak is present in 'ModScript'):
            Locate the NMS PCBANKS pak containing the .MBIN and extract it to 'MODBUILDER\MOD'
            
        * Each .MBIN is passed to the appropriate MBINCompiler.exe to decompile it to an .EXML

    - Note: If some lua scripts reference (individually or collectively) ALL .MBINs in a pak in 'ModScript',
            then all the .MBINs 'of the pak' will end up in the final mod pak(s)
          
            In this case only, this means the original pak will not be needed with the created pak(s) in 'PCBANKS\MODS'
      
            Otherwise, as a rule, a pak with some script in 'ModScript' should be used in 'PCBANKS\MODS'
      
            Do not delete that original pak 'forever' because the changes 
            in those .MBINs may not have been 're-created' completely in the provided lua script(s),
            you may need to keep it, as a reference, to re-create the mod pak(s) after a future NMS update.
